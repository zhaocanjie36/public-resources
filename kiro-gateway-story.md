# Kiro Gateway Manager 开发故事

## 项目背景

2026年3月2日，我们启动了一个全新的项目：Kiro Gateway Manager。这是一个基于 Web 的多账号 Kiro Gateway 管理系统，目标是让管理几百个 GitHub 账号的 Kiro Gateway 实例变得简单高效。

## 为什么需要这个系统？

Kiro 是一个强大的 AI 编码助手，但当你需要管理大量 GitHub 账号时，手动管理每个 Gateway 实例会变得非常繁琐。我们需要一个统一的管理平台，能够：

1. 批量管理几百个 GitHub 账号
2. 自动化 Gateway 的启动、停止和重启
3. 实时监控系统资源和账号状态
4. 智能分配端口，避免冲突
5. 追踪使用统计，优化资源利用

## 技术架构

我们选择了现代化的技术栈：

**后端：FastAPI**
- 高性能的 Python Web 框架
- 异步支持，处理并发请求
- 自动生成 API 文档
- SQLite 数据库存储账号信息

**前端：React + Tailwind CSS**
- 现代化的组件化开发
- 响应式设计，适配各种屏幕
- 实时数据刷新，监控系统状态
- Ant Design 组件库，美观易用

**基础设施：Docker + Nginx**
- Docker 容器化部署 Gateway 实例
- Nginx 反向代理，统一入口
- systemd 服务管理，开机自启

- Docker 容器自动管理
- 智能端口分配算法

**智能端口分配**
这是一个亮点功能。我们设计了一个算法，端口范围是 8001-8100，但不是简单的顺序分配，而是优先填补空缺。比如：
- 如果 8001、8003、8005 已占用
- 下一个分配的是 8002（填补空缺）
- 而不是 8006（顺序分配）

这样可以最大化端口利用率，避免浪费。

### 第二天：问题修复与优化（3月3日凌晨）

**SSH 连接问题**
凌晨4点多，Boss 遇到了 SSH 连接问题。开启 Shadowrocket VPN 后无法连接服务器，但 ping 正常。

经过排查，我们发现：
- VPN 在系统路由表层面拦截了 Tailscale 流量
- 应用层配置（SSH config、环境变量）无法绕过
- 解决方案：修复路由表规则

```bash
sudo route delete -net 100.64.0.0/10
sudo route add -net 100.64.0.0/10 -interface utun4
```

这个问题让我们学到：VPN 模式在网络层拦截，需要在路由层面解决。

**Docker 权限问题**
部署后发现 Docker 容器无法写入数据库文件。原因是：
- 容器内使用 root 用户
- 挂载的目录权限不足
- 只读挂载标志导致无法写入

解决方案：
1. 删除只读挂载标志
2. 自动设置目录权限为 777
3. 确保容器可以正常读写

### 第二天上午：UI 功能优化（3月3日 5:26-5:40）

- 删除"在新窗口打开"按钮
- 只保留"复制链接"按钮

### 第二天上午：Bug 修复（3月3日 9:16-9:31）

**问题1：API Key 显示错误**
- 前端显示固定密码，而不是真实的随机 API Key
- 原因：使用了硬编码的常量
- 解决：从账号数据中读取真实的 api_key 字段

**问题2：OpenClaw 切换失败**
- 切换账号时报错：找不到 openclaw 命令
- 原因：systemd 服务的 PATH 中没有该命令
- 解决：使用 systemctl --user restart openclaw-gateway
- 需要设置 XDG_RUNTIME_DIR=/run/user/0 环境变量
